# Claude Code 方案与代码审查清单

## 审查优先级

1. 是否违反项目铁律。
2. 是否服务于跨患者泛化或 Phase 3 预后主线。
3. 是否有可验证的实验设计，而不是只增加复杂度。
4. 是否控制训练成本和 token 成本。
5. 是否能在本地冒烟、远程完整训练、结果回传之间顺畅衔接。

## 方案审查

看到 Claude Code 的方案时，优先问：

- 该方案解决的是当前主线问题吗：跨患者泛化、Phase 3 预后、9 患者扩展、服务器训练？
- 是否已有负向经验：OmiCLIP 跨患者、AttnPool 跨患者、EGN-v1、过参数化 token 方案？
- 是否有对照组：至少包含当前最佳或合理基线。
- 是否有停止条件：冒烟失败、val_loss 不收敛、PCC/AUC 不提升、训练成本过高。
- 是否先做小实验：1 epoch 冒烟 -> 单 fold/单患者快速验证 -> 3 折或远程长训。

## 代码审查

高风险点：

- 修改了 `histogene/`、`egnv1/`、`egnv2/`。
- 硬编码 `D:\AI...`、服务器路径或患者路径。
- 把 `patch_noov_spilt` 改成 `split`。
- 用 `val_pcc` 选择最佳 checkpoint。
- 写出的 `predictions.csv` 列名不兼容可视化。
- 新增训练参数但没有默认回退到旧行为。
- 修改数据交集逻辑导致样本数异常减少。
- 在 Windows 命令中遗漏 `PYTHONIOENCODING=utf-8`。
- 用 PowerShell `&&` 或混用 Bash/PowerShell 语法。
- 覆盖历史结果目录，而不是创建带时间戳的新输出。

## 运行结果审查

优先检查：

- 样本数是否符合预期，是否突然从二层交集退回三层交集导致数据变少。
- loss 是否正常下降，是否出现 NaN/Inf。
- evaluate 是否同时输出 PCC、MAE、R2。
- checkpoint 选择依据是否是 `val_loss`。
- 单患者提升是否能迁移到跨患者，不能只看 HYZ15040。
- 结果是否更新到排名文件或实验日志。

## 给 Claude 的纠偏风格

- 先指出最高风险的一处偏航，再给最短修复路径。
- 能给命令就给命令，能定位文件就定位文件。
- 不要求 Claude 大面积重构，优先小补丁、小验证。
- 对“看起来更复杂”的模型保持警惕，要求先过跨患者小实验。

## Token 节约策略

- 默认只读 `codex_memory/` + 相关 diff，不全量读仓库。
- 审查代码时先看 `git diff --stat` 和变更文件列表。
- 只打开与风险相关的文件和函数。
- 训练日志只看开头配置、样本数、前 1-2 epoch、最佳指标和异常栈。
- 长文档只读取标题、摘要、结论和约束段。
