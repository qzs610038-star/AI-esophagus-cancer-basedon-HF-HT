## 2. UNI2-h 在线 LoRA 渐进解冻——启动

### 2.1 动机

DenseNet121 CLS 头方案（三折均值 0.3969）在此前多次验证中已确认优于基础 Token 头方案（三折均值 0.3812），两者均使用 UNI2-h 作为 frozen backbone，差异仅在下游头架构和特征提取方式（CLS vs Tokens）。但"完全冻结 backbone、仅训练下游 MLP"的天花板正在逼近——Fold2 仅 1 epoch 即过拟合，说明 frozen 特征的信息量已不足以支撑进一步泛化。

**在线 LoRA（LoRA + 渐进解冻）** 是当前最有希望突破 frozen 瓶颈的方向：
- 不改模型架构，通过低秩适配器让 300M 参数的 backbone 适应食管癌病理数据
- 仅 ~1.77M LoRA 参数可训练（backbone 的 0.6%），过拟合风险可控
- 三阶段渐进设计（LoRA → 末 2 层解冻 → 末 4 层解冻），逐步"松开枷锁"

### 2.2 三阶段架构速览

```
Stage 1 (LoRA):    全 24 层注入 LoRA (rank=8, qkv+proj)
                   可训练: ~1.77M LoRA + 4.0M 下游头 = 5.77M
                   
Stage 2:           末 2 层(22-23) LoRA merge + 全解冻 + 0-21 层 LoRA
                   可训练: ~3.5M LoRA + ~28M 解冻 + 4.0M 下游头

Stage 3:           末 4 层(20-23) LoRA merge + 全解冻 + 0-19 层 LoRA
                   可训练: ~3.5M LoRA + ~56M 解冻 + 4.0M 下游头
```

**关键设计**：
- **差分学习率**：LoRA `1e-4`，解冻 backbone `1e-5`（慢 10 倍，保护预训练权重），下游头 `1e-4`
- **LoRA merge**：跨阶段迁移时将 LoRA 学到的 ΔW 烧录进原始权重（`W_new = W_orig + B×A × α/r`），确保前一阶段知识不丢失
- **混合精度 + 梯度累积**：batch_size=1 × grad_accum=2，适配 RTX 4060 8GB 显存

## 6. 本周经验教训

1. **LoRA 1 epoch 即超 frozen 基线**——backbone 存在明确的任务适配空间，frozen 特征瓶颈非理论假设而是可观测现实
2. **但过拟合速度远超预期**——Epoch 2 即达最优，Epoch 4 跌破 frozen 基线。2390 单患者样本完全不足以支撑 1.77M LoRA 参数的有效训练，参数/样本比 ~740:1
3. **在线训练的正则化不能"以后再做"**——frozen 训练的 AugMix + TV Loss 经验必须尽快迁移至在线管线，否则 LoRA 的潜力被过拟合完全掩盖
4. **在线 LoRA 训练在 RTX 4060 8GB 上可行**——batch_size=1 + grad_accum=2 + AMP，显存 4.5GB，安全余量充足。10 epoch 仅 43 分钟
5. **Train-Val Gap 是最有用的监控指标**——单看 Val PCC 可能错过过拟合早期信号（Epoch 2→3 时 Val PCC 仅降 0.008，但 Gap 已从 0.05 扩至 0.12）
6. **Python `-u` 参数至关重要**——后台训练脚本必须加 `python -u` 才能实时看到 epoch 日志
7. **HF 离线加载需要 `HF_HUB_OFFLINE=1`**——网络不可达时 HF client 默认重试 5 次（~30s），模型仍从本地缓存加载但浪费时间

## 附录 D：预判专业提问与回答

### Q1：UNI2-h LoRA 1 epoch 就超 frozen 基线，是不是说明 frozen 基线训练不充分？

**回答要点**：
1. 不是——frozen 基线（0.5227）是在 DenseNet121 MLP 上训练了完整 15 epoch（早停于 epoch 5），下游头已充分收敛
2. LoRA 的优势在于端到端——backbone 看到任务数据后，即使仅 1 epoch，其 CLS token 输出的"任务相关特征"已经优于 frozen backbone 的"通用病理特征"
3. 这从反面证明了 frozen 瓶颈的真实存在：backbone 特征再强，没看过食管癌数据就会受限

### Q2：LoRA 训练需要在线跑 backbone forward+backward，RTX 4060 8GB 够用吗？

**回答要点**：
1. 已验证足够：batch_size=1 + grad_accum=2 + AMP，GPU 显存 4.5GB/8.2GB，余量充足
2. 单 epoch ~255 秒，10 epoch ~42 分钟，本地短训完全可行
3. 跨患者 Fold1（~14K 样本）每 epoch ~25 分钟，15 epoch ~6 小时，建议服务器 RTX 4080
4. Stage 2/3 解冻末层会增加显存（需存储更多参数的梯度），届时可能需要服务器

### Q3：为什么不直接用 PEFT 库的 LoRA，要手写？

**回答要点**：
1. UNI2-H 使用 timm 自定义 ViT 结构，`attn.qkv` 是融合的 Linear（1536→4608），而非 HuggingFace 标准的 `q_proj/k_proj/v_proj` 三独立模块，PEFT 库的命名规则不匹配
2. 手写实现可以精确控制注入目标（qkv+proj），也避免 PEFT 库依赖
3. LoRA 核心逻辑仅 ~80 行（`LoRALinear` 类），复杂度不高，自检已验证正确性（merge 前后输出差异=0）

### Q4：online CLS 和 online Tokens 两个模型有什么区别？优先跑哪个？

**回答要点**：
1. CLS 模式：backbone → [CLS] token → MLP 回归头。Tokens 模式：backbone → 65 tokens → LightweightTokenEncoder（2 层 Transformer）→ mean pool → MLP 回归头
2. 参考离线实验：UNI Tokens（保留 65 token 空间信息）优于 UNI CLS（全局特征池化）。但有坐标编码的 Token 模型显存更大
3. 优先跑 CLS 模式——它更轻量，先验证 LoRA 方向正确性。若有效再升级到 Token 模式

### E.1 背景与定位

5月25日（周日）完成了 RL 方法在 PFMval 项目中的适用性系统分析。核心问题：在当前数据约束下，RL 能做什么、该怎么做。

确立双轨策略：
- **方案 B（DQN 训练调度器）**：训练 Agent 动态调整 LR + TV weight，选 DQN（off-policy 可复用历史 CSV 离线训练）
- **方案 D（BO 通路选择）**：Bayesian Optimization 搜索对预后最优的通路子集，选 BO（组合优化比序贯 DQN 更匹配问题结构）

Codex 审查（5.30）已将 RL 方案 B 降级为 P2（暂缓），方案 D 维持绑定 Phase 3 AUC baseline。

### E.2 关键要点

| | 方案 B | 方案 D |
|------|------|------|
| **做什么** | RL 调度器动态调整训练超参 | 搜索预后预测最优通路子集 |
| **方法** | DQN | Bayesian Optimization |
