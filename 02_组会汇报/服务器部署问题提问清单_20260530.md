## 一、我们想实现什么？（愿景）

### 目标工作流

```
本地编写代码 → 一键推送 → 服务器自动训练 → 结果自动回传
```

**具体来说**：在本地电脑上执行一条命令，代码自动同步到 RTX 4080 服务器，GPU 训练启动，训练完成后结果（checkpoint、predictions.csv、训练曲线）自动回传本地。

### 为什么需要这个？

| 现状（RDP 手动） | 目标（自动化） |
|:---|:---|
| 每次训练：手动打包代码 → RDP 登录 → 复制文件 → 敲命令 → 等完成后 RDP 登录回收结果 | 一行命令：`./deploy/run.sh 'python train_xxx.py --epochs 150'` |
| 三折 CV 需三次手动操作，每折 5-10h，跨数天 | 三个命令发完即可，结果自动回收 |
| 无法批量实验（扫参、消融需要几十组训练） | 脚本化批量提交，GPU 空闲自动跑下一组 |
| 人必须记得回收结果，容易遗忘 | 训练完成自动通知 + 回传 |
| 不适合长期迭代（本项目预计持续数月） | 可持续的 GPU 计算管线 |

### 我们已经准备好了什么？

整套自动化工具链**代码已全部就绪**，只差网络通道这一步：

| 组件 | 状态 | 说明 |
|------|:---:|------|
| `deploy/push.sh` | ✅ 就绪 | Git 方式推送代码到服务器 |
| `deploy/pull.sh` | ✅ 就绪 | 从服务器拉取训练结果 |
| `deploy/run.sh` | ✅ 就绪 | 推送+训练+回传一条命令 |
| `deploy/cmd_server.py` | ✅ 就绪 | 服务器端 HTTP 命令服务（Token 鉴权 + 命令白名单） |
| `deploy/cmd_client.py` | ✅ 就绪 | 本地客户端，远程执行训练命令 |
| `deploy/config.sh` | ✅ 就绪 | 服务器连接配置 |

**换句话说**：只要网络能通，5 分钟内就能从"手动 RDP"切换到"一键自动训练"。

## 二、尝试过程中遇到了什么困难？

### 2.1 服务器网络环境实测

| 目标 | 状态 | 说明 |
|------|:---:|------|
| baidu.com | ✅ | 国内网络正常 |
| download.pytorch.org | ✅ | 下载速度 14.2 MB/s，很快 |
| hf-mirror.com | ✅ | HuggingFace 镜像可达 |
| trycloudflare.com | ✅ | Cloudflare 可达 |
| api.cloudflare.com | ✅ | Cloudflare API 可达 |
| **huggingface.co** | ❌ | HuggingFace 官方站不可达 |
| **github.com** | ❌ | GitHub 不可达 |
| **bore.pub** | ❌ | 反向隧道服务不可达 |

**结论**：服务器能访问国内站点和部分国际 CDN，但 GitHub 和部分国外服务被封。这对我们有两个影响：
- ✅ PyTorch 等可从国内镜像安装（已验证）
- ❌ GitHub 下载工具（bore、cloudflared）受阻
- ❌ HuggingFace 门控模型（UNI2-h）无法直接下载

### 2.2 四种网络通道方案，全部受阻于同一根因

| 方案 | 原理 | 失败原因 |
|------|------|----------|
| **SSH 直连** | 服务器开放 SSH 端口，本地直接连接 | 端口 22 被 WSL2/Ubuntu 占用；无权限修改配置 |
| **bore 反向隧道** | 服务器出站连接 bore.pub，本地通过 bore.pub 中转 | bore.pub 不可达（timeout） |
| **Cloudflare Tunnel** | 服务器出站连接 Cloudflare，本地通过 Cloudflare 中转 | cloudflared.exe 需从 GitHub 下载 → GitHub 不可达 |
| **HTTP 命令服务（自建）** | 服务器监听 8080，本地直接 HTTP 调用 | 服务已启动运行 ✅，但防火墙无入站规则，外网无法访问 |

**共同根因**：新账号 AIPatho1 **没有管理员权限**，无法：
- 添加防火墙入站规则（哪怕只开一个端口）
- 安装系统级网络工具（Tailscale、cloudflared）
- 修改 sshd_config 或重启 sshd 服务

|------|:---:|
| 服务器环境验证（RTX 4080 + CUDA 12.9） | ✅ |

## 三、需要请求管理员协助的事项

### 🔴 紧急（当前阻塞部署）

**请求 1：跨用户文件复制**

```
xcopy "C:\Users\AIPatho\.cache\huggingface" "C:\Users\AIPatho1\.cache\huggingface\" /E /I /H /Y
```

- **原因**：UNI2-h 病理基础模型（~3GB）已在旧用户 AIPatho 目录下缓存，新用户 AIPatho1 无权读取（Windows 用户目录权限隔离）
- **影响**：不复制则无法加载模型，训练完全不能启动
- **耗时**：约 1 分钟（纯本地文件复制）
- **备选方案**：给 AIPatho1 授予访问 `C:\Users\AIPatho\.cache\` 的读取权限

### 🟡 重要（解锁自动化训练管线）

**请求 2：防火墙入站端口开放（三选一）**

| 选项 | 端口 | 效果 |
|------|------|------|
| A | 开放 **22330** | SSH 直连，最稳定安全的方案 |
| B | 开放 **8080** | HTTP 命令服务直接可用（零额外部署） |
| C | 开放**任意端口** | 我们适配即可 |

只需开放**一个端口**，就能从"每次 RDP 手动操作"变成"本地一键训练"。

**请求 3：Tailscale / ZeroTier 安装**

如果开放端口不可行，安装 Tailscale 是**零端口开放**的替代方案：
- 不需要任何入站防火墙规则
- 只需服务器能出站连接（已验证 `tailscale.com` 可达）
- 免费版支持 3 用户
- 安装后本地和服务器处于同一虚拟内网，SSH + HTTP 服务直接可用

> 安装命令（需管理员权限）：`winget install tailscale.tailscale` 或从 `tailscale.com` 下载

### 🟢 建议确认

**请求 4：授予 AIPatho1 本地管理员权限**

如果能给管理员权限，上述所有问题我们可以自行解决（添加防火墙规则、安装工具、配置文件权限），不再需要反复麻烦管理员。

## 四、新旧账号数据隔离问题分析

### 4.1 账号现状

> **背景**：之前仅有一个账号，两位同学同时操作时互相打断训练进程。因此创建新账号 AIPatho1 实现用户隔离，各自独立训练互不干扰。**两个账号长期并存，不会合并或迁移。**

| 项目 | AIPatho（学长） | AIPatho1（我们） |
|------|------|------|
| 状态 | **活跃使用中** | 部署中 |
| 用户目录 | `C:\Users\AIPatho\` | `C:\Users\AIPatho1\` |
| HuggingFace 模型缓存 | ~3GB，含 UNI2-h | **空**（且无法从 HF 下载，因为 UNI2-h 是门控模型） |
| 项目代码 | `D:\AIPatho\yzq\PFMval\`（LOO 验证） | `D:\AIPatho\qzs\pfmval_deploy\`（新版三折 CV） |
| 训练数据 | `D:\AIPatho\yzq\data\` | `D:\AIPatho\qzs\data-phase2\` |
| Python 环境 | 未知 | `C:\Users\AIPatho1\pfmval_env`（新建） |

### 4.2 已验证的隔离问题

**问题 1：用户目录硬隔离（Windows 安全机制）**

AIPatho1 无法读取 `C:\Users\AIPatho\` 下的**任何**文件（PermissionError: WinError 5）。这意味着旧用户下载的所有模型权重、HF 缓存、pip 缓存均无法复用。

**实际影响**：
- UNI2-h 模型无法加载 → **训练完全阻塞**（当前状态）
- 未来如需 Virchow2、OmiCLIP 等模型，同样需要重新下载或管理员复制

**问题 2：门控模型下载困境**

UNI2-h（MahmoodLab/UNI2-h）是 HuggingFace 门控模型，需要申请授权才能下载。服务器上：
- `huggingface.co` 不可达 → 无法用官方渠道下载
- `hf-mirror.com` 可达但返回 403 → 镜像站不支持门控模型授权验证
- **唯一可行的方式**：复用旧用户 AIPatho 已下载的缓存文件

**问题 3：D 盘共享目录（影响待确认）**

| 路径 | 新账号可读？ | 新账号可写？ | 风险 |
|------|:---:|:---:|------|
| `D:\AIPatho\qzs\data-phase2\` | ✅ | ✅ | 无风险 |
| `D:\AIPatho\yzq\data\` | 待确认 | 待确认 | 如需用旧数据需确认 |
| `D:\AIPatho\yzq\PFMval\` | 待确认 | — | 仅参考旧代码，无需写入 |

### 4.3 两账号并存需协调的问题

> 旧账号 AIPatho 仍在活跃使用（学长），两个账号将长期并存。

**🧩 需要建立共享机制的问题**

| 问题 | 现状 | 建议 |
|------|------|------|
| **HF 模型缓存** | 各用户独立 `~/.cache/huggingface/`，同一模型下两份（~3GB×N），浪费磁盘 | 设置 `HF_HOME=D:\AIPatho\shared\.cache\huggingface` 两账号共用，避免重复下载 |
| **大文件模型权重** | UNI2-h 等门控模型新账号无法独立下载 | 管理员复制一次当前缓存；长期建立 D 盘公共缓存目录 |
| **共享数据访问** | 旧版数据在 `D:\AIPatho\yzq\data\`，新版在 `D:\AIPatho\qzs\data-phase2\`，当前不互通 | 如需互通，将数据放到两账号都可读写的公共目录 |

**⏱️ 需要协调使用时间的问题**

| 资源 | 限制 | 建议 |
|------|------|------|
| **GPU 显存 (16GB)** | 同时跑两个训练任务可能 OOM | `nvidia-smi` 监控 + 错峰使用，或事先沟通时间窗口 |
| **GPU 算力** | 无显式限制，但共享时各自变慢 | 非紧急训练可设低优先级 |
| **端口分配** | 每账号的网络服务需独立端口 | AIPatho 用 8080/22330，AIPatho1 用 8081/22331 |
| CPU/RAM (1TB) | 充足，非瓶颈 | — |
| 磁盘 I/O | 同时读写大量 patch 小文件 | 数据目录已天然隔离（分属不同子目录） |

**✅ 已经天然隔离、无需协调的**

- Python 虚拟环境（各自独立 venv）
- 项目代码（分属不同目录）
- 训练 checkpoint 和结果（分属不同目录）
- Windows 用户配置文件

### 4.4 建议的共存机制

```
短期（本周）：管理员复制 HF 缓存（解决当前阻塞）
             xcopy C:\Users\AIPatho\.cache\huggingface → AIPatho1

中期（6月）： 建立 D:\AIPatho\shared\ 公共资源目录
             配置两账号 HF_HOME 指向公共目录
             约定 GPU 使用时间窗口 + 端口分配

长期：       两个账号各自独立开发训练，共享公共模型缓存
             如有第三位同学加入，同样模式扩展即可
```
