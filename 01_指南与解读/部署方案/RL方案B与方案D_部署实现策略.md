## 2. 方案 B：RL 训练调度器

### 2.1 为什么选 DQN

| 问题特征 | DQN 匹配度 | 详细理由 |
|------|:---:|------|
| 离散动作空间（9 种 LR×TV 组合） | ✅ | DQN 对离散动作天然高效，输出层直接是 9 个 Q 值 |
| 状态空间小（~10 维） | ✅ | 小 MLP（3-4 层）即可建模，无需复杂网络架构 |
| 单次评估极贵（~3h GPU/次训练） | ✅ | **这是选 DQN 最关键的原因**：off-policy 允许从 ~30 份历史 CSV 构建 replay buffer 离线训练，无需在线交互 |
| Reward 噪声大（val PCC ±0.008/epoch） | ✅ | 经验回放 buffer 天然平滑噪声——随机采样打破时间相关，等效于对 reward 做多次采样平均 |
| 信用分配长距（200 epoch = 200 步） | ✅ | Double DQN 变体减少 Q 值高估，缓解长距信用分配中的误差累积 |
| 通路集变化后需重训 | 🟡 | 动作空间（LR+TV）与通路无关 → 通路集变更不影响 Agent，但最优调度可能因任务难度不同而变化 |

#### 为什么不选其他方法

| 方法 | 淘汰理由 |
|------|------|
| **PPO** | On-policy——每次策略更新后旧数据作废。训练一次 = 3h GPU，PPO 至少需要几百次在线交互 → 不可承受 |
| **REINFORCE** | 同 PPO + 收敛更慢 |
| **SAC** | 为连续动作设计。方案 B 的动作是离散的（9 种 LR×TV 组合），离散化 SAC 失去其优势 |
| **BO** | 假设每次评估独立，不能建模 epoch 间的状态转移。选"epoch 5 的 TV weight"和"epoch 6 的 TV weight"在 BO 看来是两个独立决策，丢失了序贯依赖 |
| **遗传算法** | 同上，不建模序贯依赖。且"一次完整训练"作为 fitness 评估太贵 |

### 2.2 MDP 设计

#### State（状态空间）

Agent 在每个决策点观测以下特征（~10 维向量）：

| 特征 | 维度 | 说明 |
|------|:---:|------|
| val_loss 滑动斜率 | 1 | 最近 5 epoch val_loss 的线性回归斜率 |
| val_pcc 滑动均值 | 1 | 最近 5 epoch val_pcc 的均值 |
| val_pcc 滑动标准差 | 1 | 最近 5 epoch val_pcc 的标准差（衡量稳定程度） |
| 当前 LR | 1 | log 尺度归一化（如 log10(lr)） |
| 当前 TV weight | 1 | 归一化到 [0, 1] |
| epoch 编号 | 1 | 归一化到 [0, 1]，帮助 Agent 感知训练阶段 |
| val_pcc 相对历史最高 | 1 | 当前 val_pcc / 历史最高 val_pcc——早期 >1，后期 <1 提示过拟合 |
| train_loss - val_loss gap | 1 | 过拟合程度的实时指标 |

> 窗口大小选 5 epoch 的理由：5 epoch 滑动平均将 val_pcc 噪声从 ±0.008 降至 ~±0.004（1/√5），同时保持对真实趋势的响应速度。

#### Action（动作空间）

3 种 LR 调整 × 3 种 TV weight 调整 = **9 种离散动作**：

| 动作编号 | LR 调整 | TV weight 调整 | 语义 |
|:---:|------|------|------|
| 0 | ×0.5 | ×0.8 | 双降：减速 + 减轻正则 |
| 1 | ×0.5 | ×1.0 | 降速保持正则 |
| 2 | ×0.5 | ×1.25 | 降速 + 增强正则（防过拟合） |
| 3 | ×1.0 | ×0.8 | 保持速度减轻正则 |
| 4 | ×1.0 | ×1.0 | 不变 |
| 5 | ×1.0 | ×1.25 | 保持速度增强正则 |
| 6 | ×2.0 | ×0.8 | 加速 + 减轻正则 |
| 7 | ×2.0 | ×1.0 | 加速保持正则 |
| 8 | ×2.0 | ×1.25 | 双增：加速 + 增强正则 |

**约束**：LR 硬限制在 [1e-6, 1e-3]，TV weight 硬限制在 [0.001, 0.5]，防止 Agent 探索危险区域。

#### Reward（奖励函数）

```
Reward_t = Δval_pcc_5ep_ma + α × (-Δval_loss_5ep_ma)
```

| 项 | 含义 | 说明 |
|------|------|------|
| Δval_pcc_5ep_ma | 最近 5 epoch val_pcc 滑动均值的变化 | 主目标：预测精度提升 |
| -Δval_loss_5ep_ma | val_loss 5 epoch 滑动均值的变化（取负） | 辅助目标：loss 下降也是好事 |
| α | 平衡系数，默认 0.1 | val_loss 变化量级远大于 val_pcc，需要降权 |

**决策频率**：每 5 个 epoch 做一次决策（而非每 epoch）。理由：
- 5 epoch 聚合使 SNR 从 0.125 提升至 ~0.28
- 200 epoch 训练 → 40 个决策点，对 DQN 来说步数合理

### 2.3 离线训练策略

#### 阶段一：构建 Replay Buffer

```
输入：~30 份历史 training_history_*.csv

对每份 CSV：
  for ep in range(5, len(csv), 5):  # 每5 epoch一个决策点
      state  = 提取 ep-4 ~ ep 的滑动统计量 + 当前超参 + epoch编号
      action = 从 CSV 推断当时实际使用的超参变化（可能均为"不变"）
      reward = val_pcc_5MA(ep+5) - val_pcc_5MA(ep)
      next_state = 下一个决策点的观测
      存入 replay buffer

输出：replay buffer 约 30×8 = 240 条 transition
```

**关键问题——历史数据中缺少 action 多样性**：
- 大多数历史训练用的都是固定 schedule，action 始终是"不变"
- Agent 从这些数据里只能学会"不动最好"——因为历史数据里"动了"的后果它没见过

**缓解方案**：
1. **数据增强**：对每条真实 transition，制造"如果当时做了不同选择会怎样"的反事实样本。但这不可行——我们无法重跑历史。
2. **TV Sweep 数据利用**：9 份 TV sweep 数据中，不同 run 用了不同的固定 TV weight，可以构造"跨越不同 run 的伪 transition"——如 TV w=0.01 的 run 在 epoch 10 的状态，接上 TV w=0.1 的 run 在 epoch 15 的状态（跨 run 拼接）。这假设了"TV weight 对训练轨迹的影响是相对平滑的"。
3. **ε-greedy 在线微调**：离线预训练后，在真实训练中开启 ε-greedy 探索（ε 从 0.3 衰减到 0.05），用在线数据逐步修正离线学到的保守策略。
4. **降低对 PCC 增益的期望**：这进一步支持了"方案 B 是方法验证"的定位。

#### 阶段二：训练 DQN Agent

```
DQN 架构：
  Input:  state向量 (dim=10)
  Hidden: 3层 MLP (64→128→64) + ReLU
  Output: Q值向量 (dim=9)

训练配置：
  - Replay Buffer 容量: 10,000
  - Batch Size: 32
  - 目标网络更新频率: 每 10 个训练步
  - γ (折扣因子): 0.95
  - 优化器: Adam, lr=1e-4
  - 使用 Double DQN (防 Q 值高估)
```

#### 阶段三：在线验证

1. 在 HisToGene-UNI-Tokens + AugMix + TV 基础上，用 DQN Agent 控制训练 1 个 Fold
2. 对比固定 schedule 的 val_pcc 和 val_loss 曲线
3. 判定 Agent 行为是否合理（而非只看最终数字）：
   - ✅ 是否在 val_pcc 震荡时主动降 LR
   - ✅ 是否在过拟合出现时（train-val loss gap 增大）增大 TV weight
   - ✅ 是否在 plateau 期尝试 warm restart（升 LR）
   - ❌ 如果 Agent 学到的策略是"什么都不动"，说明离线数据缺乏行为多样性

### 2.4 成功标准

| 层级 | 标准 | 判定方式 |
|------|------|------|
| 最低 | DQN Agent 学到的策略非平凡（非"永远不动"） | 查看 action 分布，至少 3 种动作被使用 |
| 合格 | Agent 调度曲线 vs 固定调度：val_loss 不差于固定调度 | 比较 val_loss 最低点 |
| 良好 | Agent 能复现"早期轻正则→后期强正则"的阶段策略 | 查看 TV weight 随 epoch 的变化趋势 |
| 优秀 | 三折评估 PCC > 固定调度 ≥ +0.005 | 统计检验 p<0.05 |

## 3. 方案 D：RL 通路选择

### 3.1 为什么选 Bayesian Optimization

| 问题特征 | BO 匹配度 | 详细理由 |
|------|:---:|------|
| 非序贯决策（选子集→评 AUC，子集间无时序依赖） | ✅ | **这是不选 DQN 的核心原因**。强行把子集选择建模为"一条一条选通路"的序贯过程，会导致稀疏 reward（最后一步才有 AUC 信号），DQN 几乎学不到 |
| Reward 有噪声（AUC 95% CI ≈ ±0.07） | ✅ | BO 的高斯过程 (GP) 显式建模预测不确定性——"这个子集 AUC 可能是 0.75，但我只有 60% 把握" vs "这个子集 AUC 可能是 0.73，但我有 95% 把握" |
| 评估便宜（预计算后秒级） | ✅ | BO 可以用几十到几百次评估找到好解，完全够用 |
| 需要可解释性 | ✅ | GP 的核函数超参数可直接解读为"每条通路的独立贡献"和"通路间交互强度" |
| 组合空间大（2^30 ≈ 10^9） | ✅ | BO 的采集函数 (Expected Improvement / UCB) 在 exploration vs exploitation 之间高效平衡，远优于随机搜索或网格搜索 |

#### 为什么不选其他方法

| 方法 | 淘汰理由 |
|------|------|
| **DQN** | 如果序贯建模为"每一步加/删一条通路，最后一步给 reward"→ 稀疏 reward + 长 horizon + 动作顺序无关（ABC 和 CBA 是同一子集）→ 状态空间爆炸且无效 |
| **PPO/REINFORCE** | 同上 + on-policy 白浪费评估预算 |
| **遗传算法** | **可以用的替代方案**，且更简单。缺点是不如 BO 有不确定性量化（GA 只说"这个好"，不说"我不确定"），且对连续 reward 的利用效率低于 BO。如果 BO 实现复杂度是障碍，GA 是合理退路 |
| **Thompson Sampling** | 假设每条通路独立（多臂老虎机），无法捕获通路间交互（如通路 A+B 的预后价值 ≠ A 的价值 + B 的价值） |
| **LASSO/Elastic Net** | 也可以做通路选择，且非常简单。缺点：(1) 线性假设限制了对交互效应的捕获；(2) 选出的通路是"对预后有线性贡献"而非"对预后预测最有用"（预测能力≠统计显著性）；(3) 无法利用 BO 的不确定性来指导探索。如 BO 效果不显著，LASSO 可作为 baseline 对照 |

### 3.2 MDP / 优化问题设计

> 注意：BO 不是 MDP，是 black-box optimization。此处用类似 MDP 的结构来描述其要素。

| 要素 | 设计 |
|------|------|
| **搜索空间** | 30 条通路的任意非空子集（共 2^30 - 1 ≈ 10^9） |
| **目标函数** | f(子集) = 80 患者 5-fold CV 预后 AUC |
| **优化器** | Bayesian Optimization with Gaussian Process |
| **采集函数** | Expected Improvement (EI) —— 选"期望提升最大"的下一个子集去试 |
| **核函数** | Matern 5/2 kernel —— 在组合空间中建模平滑性 |
| **初始化** | 随机采样 20 个子集评估（包含全 30 通路和单条最优通路作为 anchor） |
| **总评估预算** | 200 次评估（预计算后每次 <5 秒，总计 <20 分钟） |

### 3.3 预计算策略：工程关键

这是方案 D 从"不可行"变为"高可行"的关键。

#### 步骤一：一次性预计算（GPU，offline，~1 小时）

```
对 320 张 HE 切片：
  Stage 1 模型推理 → 30 条通路 × 320 切片 = [320, 30] 预测矩阵
  存储为 pred_matrix.npy (~75 KB)

对每张切片，额外存储：
  - 切片所属患者 ID
  - 切片在患者内的序号
```

#### 步骤二：多区域聚合（CPU，秒级）

对 RL 选中的通路子集（k 条通路）：

```
从 pred_matrix 取 k 列 → [320, k]

对每位患者 p：
  该患者的切片预测 [n_p, k]（n_p 为该患者的切片数）
  
  聚合方式（按通路分别应用）：
    mean_pooling  → 患者级特征 [k]
    max_pooling   → 患者级特征 [k]（捕获最差区域信号）
    var_pooling   → 患者级特征 [k]（捕获异质性信号）
  
  患者级特征: concat([mean_k, max_k, var_k]) → [3k]

最终: [80, 3k] 特征矩阵
```

> 聚合策略是否包含在搜索空间中：**v1 先固定为 mean+max+var 三者拼接，不做选择**。v2 可考虑让 BO 对每条通路分别选择最优聚合方式（动作空间从 2^30 扩大到 4^30，建议取得 v1 结论后再决定是否值得）。

#### 步骤三：预后评估（CPU，秒级）

```
对每条患者级特征 [80, 3k]：
  5-fold stratified CV:
    训练: Logistic Regression / Random Forest
    预测: 每折的阳性概率
  计算: 整体 AUC + 95% CI (DeLong)
  
  → Reward = AUC_mean
```

### 3.4 搜索策略细节

#### Encoding 设计

将通路子集编码为 BO 可处理的连续表示：

```
方案 A（推荐）：Binary encoding
  x = [b1, b2, ..., b30], bi ∈ {0, 1}
  用 Hamming kernel 度量子集间距离

方案 B（备选）：Pathway embedding
  x = 30 条通路在 9 患者上的 PCC 向量 PCA 降维后的坐标
  优点：相似的预测表现 → 相似的 embedding → GP 更容易建模
  缺点：丢失了"哪条通路"的身份信息
```

**v1 使用方案 A（Binary + Hamming Kernel）**，简单且可解释。

#### 搜索过程

```
1. 随机初始化 20 个通路子集 + 评估 → 初始 GP 模型
2. For t = 1 to 180:
     a. GP 模型预测所有 10^9 个子集的 mean + variance
     b. 采集函数 (EI) 选出下一个最值得评估的子集
     c. 如果子集已被评估过，选 EI 次优
     d. 评估该子集（预计算矩阵 + 聚合 + 5-fold CV）
     e. 更新 GP 模型
3. 输出：
     a. 最优子集（AUC 最高）
     b. 通路重要性排序（基于 GP 超参数 + 出现频率）
     c. AUC vs 子集大小曲线
```

#### 可解释性输出

BO 的 GP 模型可产出以下可解释性信息：

1. **通路重要性**：对每条通路，计算"包含该通路的子集 vs 不包含的平均 AUC 差异"
2. **通路协同/拮抗**：GP 核函数的交互项超参数 → "哪些通路倾向于一起被选中"
3. **最优子集大小**：AUC 随子集大小的变化曲线 → "是否少数几条通路就够了"
4. **稳健性**：Bootstrap 100 次，看哪些通路在 >80% 的 bootstrap 样本中被选中

### 3.5 备选方案：遗传算法（降低实现门槛）

如果 BO 的 GP 实现（需要 pyGPGO 或 scikit-optimize）遇到工程困难，遗传算法可作为 fallback：

```
1. 初始化 50 个随机通路子集（population）
2. 评估所有子集的 AUC（fitness）
3. For generation = 1 to 50:
     a. Tournament selection（选 fitness 最高的 20 个）
     b. Crossover（随机交换两条染色体的通路）
     c. Mutation（以 5% 概率翻转每条通路的选中状态）
     d. 评估新子集，替换 fitness 最低的个体
4. 输出：最优子集 + 通路出现频率（作为重要性排序）
```

**GA vs BO 的 trade-off**：
- GA 更容易理解和实现，天然并行
- BO 样本效率更高（同样 200 次评估，BO 通常找到更好的解），且提供不确定性量化

### 3.6 成功标准

| 层级 | 标准 | 判定方式 |
|------|------|------|
| 最低 | BO 能找到 AUC ≥ 全 30 通路的子集（非劣效） | 最优子集 AUC ≥ 全通路 AUC - 0.01 |
| 合格 | BO 选出的子集 AUC > 全通路，且子集大小 < 20 | 统计检验 p<0.05 |
| 良好 | 通路重要性排序有一定的生物学可解释性 | 排名前 5 通路在文献中有预后关联报道 |
| 优秀 | 最优子集在 9 患者 ST 数据上的预测 PCC 也不差于全 30 通路 | 证明通路选择既优化了预后也保持/提升了通路预测精度 |

## 5. 需向医生确认的事项清单

### 5.1 高优先级（影响方案设计）

| # | 问题 | 对方案的影响 | 假设的默认值 |
|:---:|------|------|------|
| **1** | **80 患者的多张 HE 切片是什么关系？**<br>选项：同肿瘤多区域 / 不同时间点 / 重复扫描 | 决定多区域聚合策略（mean/max/方差 哪个为主）。如果是不同时间点，需要用时间序列建模而非空间聚合 | 同肿瘤多区域 → 用 mean+max+var 三者拼接 |
| **2** | **80 患者 H&E 与 3/9 ST 患者 H&E 是否同一医院/扫描仪/染色协议？** | 决定 Stage 1 模型在 80 患者上的预测可信度。如果不一致，需要染色归一化或 domain adaptation，否则 BO 的 reward 信号被分布漂移污染 | 完全一致 → Stage 1 直接迁移 |

### 5.2 中优先级（影响结果解读）

| # | 问题 | 对方案的影响 | 假设的默认值 |
|:---:|------|------|------|
| **3** | 预后标签的具体定义（如 3 年 OS / 5 年 DFS，cutoff 阈值如何确定） | 影响 AUC 计算和 reward 的临床意义 | 二分类，标签已由临床确定 |
| **4** | 80 患者的事件率（好预后 vs 差预后的比例） | 事件率 <20% 或 >80% → AUC 置信区间更宽 → BO 需要更多评估次数 | 事件率 30-50%（均衡） |
| **5** | 30 条通路的生物学分组（如有已知的功能分类） | 可用于 BO 搜索的先验——如果某组通路在生物学上冗余，BO 可能只需每组选 1-2 条 | 无先验，BO 从零探索 |
| **6** | 后续预后任务是否可能从二分类变为多分类/生存回归 | 影响 reward 函数设计。生存回归用 C-index，二分类用 AUC。框架不变但需提前预留接口 | 先二分类，框架可扩展 |

## 6. 风险评估与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|:---:|:---:|------|
| 离线 DQN 数据缺乏 action 多样性 → 学到"不动最好"的平凡策略 | 高 | 中 | ε-greedy 在线微调 + 跨 run 数据增强 + 降低期望（方案 B 定位为方法验证） |
| 9 患者到齐延迟 → 方案 B 窗口拉长但方案 D 无法启动 | 中 | 高 | 方案 B 先收尾归档，方案 D 可用 3 患者先做框架验证（但 reward 不可靠） |
| 80 患者分布漂移 → 方案 D 的 reward 被污染 | 低（同一来源）| 致命 | 染色归一化预处理 + 用 9 患者 ST 上的 PCC 作为辅助 reward |
| BO 搜索过拟合 80 患者 → 选出的通路不泛化 | 中 | 高 | Nested CV：80 患者内部 5-fold，外层再做一次留出验证 + Bootstrap 稳定性分析 |
| 预计算矩阵占用过大或 AGgregation 策略选择错误 | 低 | 低 | [320,30] float32 ≈ 38KB，可忽略。多聚合策略并行评估，选 AUC 最优的 |
| Phase 3 管线延迟 | 中 | 中 | 方案 D 的 BO 框架可以先在模拟数据上开发和测试 |
